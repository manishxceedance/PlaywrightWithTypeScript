Lesson 6: Generics for Reusable Components

Level: Intermediate-Advanced Learn how to write reusable, type-safe code using TypeScript generics.

Generics let you write code that works with any type, while keeping type safety. They’re essential for building flexible utilities, data structures, and APIs.

1. What is a Generic?

A generic is a placeholder for a type. It lets you write one function, class, or type that works for many types.

// Generic function
function wrapInArray<T>(value: T): T[] {
  return [value];
}
const numbers = wrapInArray(10);      // number[]
const names = wrapInArray("Neha");   // string[]
Line-by-line explanation:
  1. function wrapInArray<T> — The <T> after the function name declares a generic type parameter. Think of T as a blank label. TypeScript fills it in automatically when you call the function based on what you pass in — you do not need to specify it yourself.
  2. (value: T): T[] — The parameter is of type T and the return type is an array of T. Both Ts refer to the same resolved type. If T resolves to number, the return type automatically becomes number[].
  3. return [value] — Wraps the single input in a new array. Simple logic, but now fully type-safe for any type you pass in — no code duplication needed.
  4. wrapInArray(10) — TypeScript sees the argument is 10 (a number), infers T = number, and the return type is number[]. No annotation needed on the call side.
  5. wrapInArray("Neha") — TypeScript infers T = string and returns string[]. The same function body handles both types. This is the core value of generics: one implementation, infinite type safety.

2. Generic Types and Interfaces

You can use generics in type aliases and interfaces to describe reusable shapes.

// Generic API response type
type ApiResponse<T> = {
  status: number;
  data: T;
  error?: string;
};

type UserPayload = { id: string; name: string };
const response: ApiResponse<UserPayload> = {
  status: 200,
  data: { id: "U1", name: "Anu" }
};
Line-by-line explanation:
  1. type ApiResponse<T> — A generic type alias. The <T> is a placeholder for whatever payload type this response will carry. You decide T when you use the type.
  2. status: number and error?: string — Fields that are the same for every response, regardless of the payload type. These never change.
  3. data: T — This field's type changes based on what T is. If T is UserPayload, then data must be a UserPayload object.
  4. type UserPayload = { id: string; name: string } — The specific shape for a user-related response. Defined separately so it can be reused in other places too.
  5. ApiResponse<UserPayload> — Fills in T with UserPayload. TypeScript now knows that response.data.id is a string. If you had written response.data.email it would fail at compile time because UserPayload has no email field.

3. Generic Constraints

Use extends to require that a generic type has certain properties.

// Only accepts values with a length property
function printLength<T extends { length: number }>(value: T): void {
  console.log(value.length);
}
printLength([1, 2, 3]); // OK
printLength("hello");   // OK
// printLength(42);     // Error
Line-by-line explanation:
  1. function printLength<T extends { length: number }> — The constraint extends { length: number } means: “only accept a T that has a length property of type number.” This is how you restrict a generic so it is not completely open to any type.
  2. (value: T): void — The function takes the constrained type and returns nothing. Declaring void as the return type makes it explicit that this function is called for its side-effect (logging), not its return value.
  3. console.log(value.length) — Safe to call because TypeScript knows T has a length. Without the constraint, TypeScript would reject this line since not all types have length.
  4. printLength([1, 2, 3]) — Arrays have a length property, so they satisfy the constraint. T resolves to number[]. Logs 3.
  5. printLength("hello") — Strings also have length. T resolves to string. Logs 5.
  6. // printLength(42) — Numbers have no length property. TypeScript rejects this at compile time: "Argument of type 'number' is not assignable to parameter of type '{ length: number }'". The error is caught before any code runs.

4. Multiple Type Parameters & Defaults

You can use more than one generic, and provide default types.

// Pair of values
type Pair<A, B = string> = { first: A; second: B };
const p1: Pair<number, boolean> = { first: 1, second: true };
const p2: Pair<number> = { first: 2, second: "default" };
Line-by-line explanation:
  1. type Pair<A, B = string> — Declares two type parameters: A (required) and B (optional, defaults to string). The = string means: if the caller does not provide a second type argument, TypeScript automatically uses string.
  2. { first: A; second: B } — The pair object has two properties whose types come from the generic parameters. Both types are set by the caller.
  3. Pair<number, boolean> — Explicitly sets A = number and B = boolean. So p1.first is a number and p1.second is a boolean. TypeScript verifies the values match.
  4. Pair<number> — Only provides one type argument. A = number and B falls back to the default string. So p2.second must be a string. Assigning a number there would be a compile error.
  5. Why defaults are useful: They reduce repetition when most callers use the same second type, while still letting callers override it when needed.

5. Generic Classes and Interfaces

Classes and interfaces can be generic too.

// Generic class for a simple stack
class Stack<T> {
  private items: T[] = [];
  push(item: T) { this.items.push(item); }
  pop(): T | undefined { return this.items.pop(); }
}
const numStack = new Stack<number>();
numStack.push(1);
numStack.push(2);
console.log(numStack.pop());
Line-by-line explanation:
  1. class Stack<T> — A generic class. The <T> is resolved when you create an instance. Every method inside the class can then use T knowing it refers to the same type throughout the class.
  2. private items: T[] = [] — An internal array that holds the stack elements. It is private so nothing outside the class can directly access or modify it. The initial value [] means an empty array at creation.
  3. push(item: T) — Adds one item of type T to the top of the stack. TypeScript prevents you from pushing the wrong type — e.g., pushing a string into a Stack<number> is a compile error.
  4. pop(): T | undefined — Removes and returns the top item. Returns undefined if the stack is empty, which is why the return type is T | undefined rather than just T.
  5. new Stack<number>() — Creates a stack that only accepts and returns number values. T is now fixed as number for this instance.
  6. numStack.pop() — Returns 2 (last item pushed). TypeScript knows the return type is number | undefined, so downstream code must handle both cases.

6. Utility Generics

Many built-in utility types use generics (like Partial, Record, etc.).

// Partial makes all fields optional
type Bug = { id: string; summary: string };
const draft: Partial<Bug> = { summary: "Crash" };

// Record creates a map
type TestResults = Record<string, "pass" | "fail">;
const results: TestResults = { "TC-1": "pass" };
Line-by-line explanation:
  1. type Bug = { id: string; summary: string } — Defines a simple two-field bug type. Used as the base for both utility type examples below.
  2. Partial<Bug> — A built-in TypeScript generic that takes every property in Bug and makes it optional (adds ? to each). So draft can have just summary, just id, both, or neither without a TypeScript error. Useful for draft forms or patch payloads.
  3. const draft: Partial<Bug> = { summary: "Crash" } — Valid because id is now optional. Without Partial, omitting id would be a type error.
  4. Record<string, "pass" | "fail"> — A built-in generic that creates an object type with string keys and values restricted to "pass" or "fail". Trying to write { "TC-1": "skip" } would be a compile error.
  5. const results: TestResults — You can now add as many test case IDs as you like, and TypeScript will verify each value is either "pass" or "fail".

7. Practical Utility: Test Data

Use generics to write flexible, type-safe utilities for test data.

// Get the first item of any array
function firstItem<T>(items: T[]): T | undefined {
  return items[0];
}
const firstCase = firstItem(["TC-1", "TC-2"]);
Line-by-line explanation:
  1. function firstItem<T>(items: T[]): T | undefined — A generic function that accepts an array of any type. The return type is T | undefined because if the array is empty, items[0] is undefined. This is safer than returning just T, which would hide the empty-array case.
  2. return items[0] — Returns the element at index 0. If the array is empty, JavaScript returns undefined here, which matches the declared return type.
  3. firstItem(["TC-1", "TC-2"]) — TypeScript infers T = string from the array elements. The return type is string | undefined. The result firstCase is "TC-1" but its type is string | undefined, reminding you to check before using string methods on it.
  4. Why this matters: If you had typed the return as just T (without undefined), calling firstCase.toUpperCase() would look safe but crash at runtime on an empty array. The union return type forces safe handling.
