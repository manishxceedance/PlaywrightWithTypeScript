Lesson 5: Classes and OOP (Object-Oriented Programming)

Level: Intermediate Learn how to build reusable, organized code using TypeScript classes and OOP features.

Classes help you build reusable framework components like page objects, API clients, and test reporters. OOP lets you organize code for real-world automation and apps.

1. What is a Class?

A class is a blueprint for creating objects with shared structure and behavior. Use classes for things that have properties and actions.

// A simple test suite class
class TestSuite {
  name: string;
  totalCases: number;

  constructor(name: string, totalCases: number) {
    this.name = name;
    this.totalCases = totalCases;
  }

  getSummary(): string {
    return `${this.name}: ${this.totalCases} cases`;
  }
}

const suite = new TestSuite("Login", 12);
console.log(suite.getSummary()); // Login: 12 cases
Line-by-line explanation:
  1. class TestSuite { — Declares a new class named TestSuite. A class is a blueprint. Every time you create one with new TestSuite(...), you get a fresh independent object with its own data.
  2. name: string; and totalCases: number; — These are class properties. Every instance of TestSuite will have its own name and totalCases. The types are declared here so TypeScript checks that you only assign the right values.
  3. constructor(name: string, totalCases: number) — A special method that runs automatically when you use new TestSuite(...). It receives the initial values and stores them on the instance using this.
  4. this.name = name — this refers to the specific object being created. Without this., you would be setting a local variable, not the object's property.
  5. getSummary(): string — A method on the class. It reads this.name and this.totalCases (the instance's own data) and returns a formatted string.
  6. new TestSuite("Login", 12) — Creates one instance of the blueprint. The constructor runs, setting name = "Login" and totalCases = 12. The result is stored in suite.
  7. suite.getSummary() — Calls the method on this specific instance. Output: "Login: 12 cases".

2. Access Modifiers

Access modifiers control where a property or method can be used.

  • public: accessible everywhere (default).
  • private: only inside this class.
  • protected: inside this class and subclasses.
class User {
  public name: string;
  private password: string;
  protected role: string;

  constructor(name: string, password: string, role: string) {
    this.name = name;
    this.password = password;
    this.role = role;
  }
}
Line-by-line explanation:
  1. public name: string — Accessible from anywhere: inside the class, in subclasses, and in code that uses the class. This is the default if you omit the modifier. Use public when the property is part of the class's intended public API.
  2. private password: string — Only accessible inside this exact class. Even a subclass cannot read it. TypeScript will error if you try user.password outside the class. Use private to hide sensitive or implementation-specific data.
  3. protected role: string — Accessible inside this class AND inside any class that extends it (a subclass). Not accessible from outside. Use protected for data that subclasses need but outside consumers should not touch.
  4. Quick rule: Start with private for everything. Change to protected only if a subclass needs it. Change to public only if outside code needs it. This discipline prevents accidental external dependencies on internal details.

3. Inheritance and super()

One class can extend another to reuse and customize behavior. Use super() to call the parent’s constructor or methods.

class BasePage {
  navigate(url: string): void {
    console.log(`Open ${url}`);
  }
}

class LoginPage extends BasePage {
  login(user: string, pass: string): void {
    console.log(`Login with ${user}`);
  }
}

const page = new LoginPage();
page.navigate("/login");
page.login("alice", "secret");
Line-by-line explanation:
  1. class BasePage — The parent class (also called superclass). It defines shared behaviour that all page classes will inherit. In a real test framework, BasePage might contain navigation, screenshot helpers, or waits.
  2. navigate(url: string): void — A method on BasePage. Any class that extends BasePage automatically inherits this method for free.
  3. class LoginPage extends BasePage — LoginPage is a child class (also called subclass). The extends keyword means it inherits everything from BasePage and can add its own methods on top.
  4. login(user: string, pass: string): void — A method that only exists on LoginPage. BasePage does not have it. This is the child adding its own specific behaviour.
  5. new LoginPage() — Creates an instance of the child class. No constructor is defined in LoginPage, so TypeScript automatically uses the parent's (which requires no arguments here).
  6. page.navigate("/login") — Calls a method that was defined in BasePage, not in LoginPage. This works because LoginPage inherited it. You get the parent's functionality without writing it again.
  7. page.login("alice", "secret") — Calls the method defined only in LoginPage. This would fail if called on a plain BasePage instance.

4. Static Members

static properties and methods belong to the class itself, not to instances. Use them for helpers or shared values.

class MathUtils {
  static PI = 3.14;
  static square(x: number) {
    return x * x;
  }
}
console.log(MathUtils.PI);
console.log(MathUtils.square(5));
Line-by-line explanation:
  1. static PI = 3.14 — A static property belongs to the class itself, not to any instance. There is only one copy of PI shared across the entire program, no matter how many MathUtils instances you create (which you would never do here anyway).
  2. static square(x: number) — A static method. Does not need this because it does not operate on instance data. It is a pure helper function that is just grouped under the class name for organisation.
  3. MathUtils.PI — Accessed directly on the class name, not on an instance. Writing new MathUtils().PI would work but is misleading and unnecessary. Static members are always accessed as ClassName.member.
  4. MathUtils.square(5) — Same pattern. Returns 25. No new MathUtils() needed.
  5. When to use static: Utility functions (math, date formatting, string helpers), factory methods, counters shared across all instances, or constants that belong conceptually to a type.

5. Abstract Classes

An abstract class can’t be created directly. It defines a contract for subclasses to implement.

abstract class Reporter {
  abstract report(message: string): void;
  log(msg: string) {
    console.log("LOG:", msg);
  }
}

class ConsoleReporter extends Reporter {
  report(message: string): void {
    console.log(`[REPORT] ${message}`);
  }
}

const rep = new ConsoleReporter();
rep.report("All tests passed");
Line-by-line explanation:
  1. abstract class Reporter — The abstract keyword prevents direct instantiation. You cannot write new Reporter(). It exists only as a base for subclasses to build on. This is TypeScript's way of defining a “must be extended” class.
  2. abstract report(message: string): void — An abstract method has no body here. It is a contract: every non-abstract subclass must provide its own implementation. TypeScript gives a compile error if a subclass does not implement all abstract methods.
  3. log(msg: string) — A concrete method with a real body. Subclasses inherit this for free and do not need to re-implement it. Mix abstract and concrete methods to share common logic while enforcing customisation points.
  4. class ConsoleReporter extends Reporter — A concrete subclass. TypeScript requires it to implement report() because the parent declared it abstract.
  5. report(message: string): void { console.log(...) } — Fulfils the abstract contract. If this method were missing, TypeScript would error: "Non-abstract class 'ConsoleReporter' does not implement inherited abstract member 'report'".
  6. new ConsoleReporter() — Works because ConsoleReporter is concrete (not abstract). The resulting instance has both report() and the inherited log() from Reporter.

6. Interface + Class Contract

Classes can implement interfaces to guarantee they provide certain methods or properties.

interface Logger {
  log(message: string): void;
}

class FileLogger implements Logger {
  log(message: string): void {
    // Write to file (simulated)
    console.log(`[FILE] ${message}`);
  }
}
Line-by-line explanation:
  1. interface Logger — Defines a contract: any class that claims to be a Logger must have a log method with this exact signature. The interface itself has no implementation code.
  2. log(message: string): void — The shape of the required method. No body here — just the signature.
  3. class FileLogger implements Logger — The implements keyword tells TypeScript: “this class promises to satisfy the Logger interface.” If log() is missing or has a different signature, TypeScript gives a compile error immediately.
  4. log(message: string): void { ... } — The concrete implementation. This class chose to simulate writing to a file by logging with a [FILE] prefix. Another class implementing Logger could write to a database, an API, or a test report instead.
  5. Why this pattern matters: Code that accepts a Logger interface can work with any class that implements it — FileLogger, ConsoleLogger, ApiLogger — without knowing which one it has. This is called programming to an interface and is fundamental to maintainable architecture.

7. Real-World Example: API Client

Combine these features to build a real-world API client class. Notice that baseUrl is protected (not private) so the child class can read it. If it were private, TypeScript would block access in SecuredApiClient.

class ApiClient {
  protected baseUrl: string; // protected: accessible in this class AND any subclass
  constructor(baseUrl: string) {
    this.baseUrl = baseUrl;
  }
  get(endpoint: string): void {
    console.log(`GET ${this.baseUrl}${endpoint}`);
  }
}

class SecuredApiClient extends ApiClient {
  private token: string; // private: only accessible inside SecuredApiClient
  constructor(baseUrl: string, token: string) {
    super(baseUrl); // calls ApiClient constructor to store baseUrl
    this.token = token;
  }
  get(endpoint: string): void {
    // this.baseUrl works here because it is 'protected', not 'private'
    console.log(`GET ${this.baseUrl}${endpoint} with token ${this.token}`);
  }
}

const api = new SecuredApiClient("https://api.example.com", "my-token");
api.get("/users");
// Output: GET https://api.example.com/users with token my-token

const api2 = new ApiClient("https://api.example.com");
api2.get("/users");
// Output: GET https://api.example.com/users
Line-by-line explanation:
  1. protected baseUrl: string — Using protected instead of private allows the child class SecuredApiClient to read this.baseUrl. Had it been private, TypeScript would report a compile error when the child tries to access it.
  2. constructor(baseUrl: string) in ApiClient — Stores the URL when the object is created. Every instance of ApiClient or its subclasses gets their own copy of baseUrl.
  3. super(baseUrl) — The first line inside any subclass constructor must call super() to run the parent's constructor logic. This is how this.baseUrl gets set for SecuredApiClient.
  4. private token: string — The token belongs only to SecuredApiClient. Nothing outside this class, not even further subclasses, can access it directly.
  5. get(endpoint: string) in SecuredApiClient — Overrides the parent method. Both have the same method name but different behaviour. This is called method overriding.
  6. this.baseUrl inside the child's get() — Works because the field is protected. If you change it to private in ApiClient, TypeScript immediately shows: "Property 'baseUrl' is private and only accessible within class 'ApiClient'".
