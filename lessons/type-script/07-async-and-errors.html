Lesson 7: Async Programming and Error Handling

Level: Advanced Learn how to write reliable async code and handle errors in TypeScript.

Most test automation and API tasks are asynchronous. Typed async code avoids fragile workflows and makes your code robust.

1. What is Asynchronous Code?

Async code lets you do things that take time (like API calls or timers) without blocking the rest of your program.

2. Promise Basics

A Promise represents a value that will be available later. Use Promise for async operations.

function waitMs(ms: number): Promise<string> {
  return new Promise((resolve) => {
    setTimeout(() => resolve(`Waited ${ms} ms`), ms);
  });
}
Explanation: This function returns a Promise that resolves after a given number of milliseconds. It uses setTimeout to wait, then calls resolve with a message. This is a basic pattern for creating asynchronous delays.

3. async/await

async functions always return a Promise. Use await to pause until a Promise is ready.

async function runHealthCheck(): Promise<void> {
  const message = await waitMs(300);
  console.log(message);
}
runHealthCheck();
Explanation: The async keyword makes the function always return a Promise. Inside, await pauses execution until waitMs(300) resolves, then logs the result. This makes async code look and behave more like synchronous code.

4. Typed API Fetch Example

Always type your async results for safety. Important: fetch is built into Node.js 18+ and all modern browsers. For TypeScript to recognise it without a "Cannot find name 'fetch'" error, add "lib": ["ES2020", "DOM"] to the compilerOptions in your tsconfig.json.

// tsconfig.json — required addition to compilerOptions:
// "lib": ["ES2020", "DOM"]
// Without this, TypeScript reports: Cannot find name 'fetch'.

type HealthResponse = { status: string; uptime: number };

async function fetchHealth(): Promise<HealthResponse> {
  const response = await fetch("/health");
  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`);
  }
  return (await response.json()) as HealthResponse;
}
Explanation:
  1. "lib": ["ES2020", "DOM"] in tsconfig — The DOM lib adds type declarations for browser and Node.js 18+ globals like fetch, Response, and Headers. Without it, TypeScript does not know these exist and rejects the code at compile time.
  2. type HealthResponse — Declares the shape of the expected JSON response. TypeScript uses this to check that you only access valid fields (e.g. health.status), catching typos like health.stats at compile time.
  3. await fetch("/health") — Sends an HTTP GET request and waits for the response object. The response body is not yet read at this point.
  4. if (!response.ok) — response.ok is true for any 2xx status code. Throwing here ensures the caller always gets a typed result or a clear error — never a silent wrong response.
  5. (await response.json()) as HealthResponse — Reads and parses the response body as JSON, then asserts it matches HealthResponse. The as cast is necessary because response.json() returns any by default.

5. Error Handling Pattern

Use try/catch to handle errors in async code. Always check the error type.

async function safeHealthCheck(): Promise<void> {
  try {
    const health = await fetchHealth();
    console.log("Service status:", health.status);
  } catch (error) {
    if (error instanceof Error) {
      console.error("Health check failed:", error.message);
    } else {
      console.error("Unknown failure", error);
    }
  }
}
Explanation: This uses try/catch to handle errors from async code. It checks if the error is an instance of Error for safe logging. This pattern prevents crashes and gives meaningful error messages.

6. Parallel Async: Promise.all

Use Promise.all to run multiple async tasks at once and wait for all results.

async function runAll() {
  const [a, b] = await Promise.all([
    waitMs(100),
    waitMs(200)
  ]);
  console.log(a, b);
}
runAll();
Explanation: Promise.all runs multiple async tasks in parallel and waits for all to finish. The results are returned as an array. This is efficient for independent async operations.

7. Utility: Safe Fetch Helper

Write a generic async helper for safe API calls. Like section 4, this uses fetch — add "lib": ["ES2020", "DOM"] to your tsconfig.json compilerOptions so TypeScript recognises it.

// Requires "lib": ["ES2020", "DOM"] in tsconfig.json compilerOptions
async function safeFetchJson<T>(url: string): Promise<T | null> {
  try {
    const res = await fetch(url);
    if (!res.ok) throw new Error("Bad response");
    return (await res.json()) as T; // explicit cast: res.json() returns 'any'; cast to T makes the return type safe
  } catch (e) {
    return null;
  }
}
Line-by-line explanation:
  1. async function safeFetchJson<T> — A generic async function. The caller decides what type T represents. For example, safeFetchJson<User>("/api/me") makes the return type Promise<User | null>.
  2. Promise<T | null> — The return type signals two possible outcomes: a correctly typed value on success, or null if anything went wrong. The caller must handle both without crashing.
  3. const res = await fetch(url) — Sends the HTTP request and waits for the response headers. The body is not yet read at this point.
  4. if (!res.ok) throw new Error("Bad response") — Fails fast for any non-2xx status. The thrown error is caught by the catch block, which returns null instead of crashing the caller.
  5. (await res.json()) as T — Reads and parses the response body as JSON, then explicitly casts it to T. This is necessary because res.json() returns any by default. Without the cast, TypeScript with strict mode may infer an implicit any return, which defeats the purpose of the generic. The cast documents your intent and enables IDE autocomplete on the result.
  6. catch (e) { return null } — Any error (network failure, JSON parse error, bad status) results in null. The caller only needs one check: if (result === null). This is much cleaner than propagating errors up the call stack for routine network issues.
