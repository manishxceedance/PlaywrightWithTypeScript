Lesson 2: Gherkin Syntax — Writing Feature Files

Level: Beginner Learn every Gherkin keyword with annotated examples.

1. The Anatomy of a Feature File

A feature file uses the .feature extension and contains one Feature block with one or more Scenarios. 

# login.feature

Feature: User Authentication
  As a registered user
  I want to log in to the application
  So that I can access my account

  Background:
    Given the application is running at "https://example.com"

  Scenario: Successful login
    Given the user is on the login page
    When the user enters email "user@test.com" and password "Pass123"
    And the user clicks the Login button
    Then the user should be redirected to the dashboard
    And the welcome message should contain "Welcome, User"

  Scenario: Failed login with wrong password
    Given the user is on the login page
    When the user enters email "user@test.com" and password "wrong"
    And the user clicks the Login button
    Then an error message "Invalid credentials" should be displayed

2. Feature

The Feature keyword starts the file and names the feature being tested. Everything below it belongs to this feature. The optional three lines below it (As a / I want / So that) are called a user story and provide context — Cucumber ignores them during execution. 

Feature: Shopping Cart Management
  As a logged-in customer
  I want to add and remove items from my cart
  So that I can control what I purchase

3. Scenario

A Scenario is one concrete test case. Each scenario is independent — it should not depend on the state left by a previous scenario. Always use present tense. 

Scenario: Adding an item to an empty cart
  Given the cart is empty
  When the user adds "Wireless Mouse" to the cart
  Then the cart should show 1 item
  And the total should be "$29.99"

4. Given / When / Then / And / But

Keyword Purpose Example
Given Precondition / initial context Given the user is logged in
When The action or event When the user clicks "Submit"
Then Expected outcome / assertion Then a success toast should appear
And Continues the previous step type And the form should be cleared
But Negative continuation But the error panel should not appear

5. Background

Background runs the same steps before every scenario in the feature file. Use it for setup that all scenarios share (e.g., navigating to a URL, logging in). 

Feature: Account Settings

  Background:
    Given the user is logged in as "admin@test.com"
    And the user is on the Account Settings page

  Scenario: Change display name
    When the user updates the name to "New Name"
    Then the profile header should show "New Name"

  Scenario: Change password
    When the user enters a new password
    Then the password should be updated successfully

Rule: Keep Background minimal. If only some scenarios need the same setup, use tags and hooks instead (covered in Lesson 4).

6. Inline Parameters (String and Number Values)

You can embed values directly in step text using double quotes. Cucumber passes these as parameters to step definitions automatically. 

When the user searches for "laptop"
And the user selects page "2"
Then "24" results should be displayed

The quoted text becomes a string parameter. Numbers can be passed unquoted and mapped to integers:

Then 24 results should be displayed

7. Docstrings (Multi-line Text)

Use triple quotes """ to pass a multi-line block of text to a step — useful for JSON payloads or long messages. 

When the user submits the following JSON body:
  """
  {
    "username": "admin",
    "role": "superuser"
  }
  """

8. Tags

Tags let you label and filter scenarios. They start with @ and sit above Feature, Scenario, or Scenario Outline. 

@smoke @login
Feature: User Authentication

  @regression
  Scenario: Successful login
    Given the user is on the login page
    ...

  @wip
  Scenario: SSO login
    Given the SSO provider is configured
    ...

Run only scenarios tagged @smoke:

npx cucumber-js --tags "@smoke"

Run smoke but not wip:

npx cucumber-js --tags "@smoke and not @wip"

9. Rule (Grouping Scenarios)

The Rule keyword groups related scenarios under one business rule inside a feature. It is optional but improves readability of large feature files. 

Feature: Password Policy

  Rule: Password must be at least 8 characters

    Scenario: Reject 7-character password
      Given the password field contains "abc1234"
      When the user submits the form
      Then an error "Password too short" should appear

    Scenario: Accept 8-character password
      Given the password field contains "abc12345"
      When the user submits the form
      Then no password errors should appear

  Rule: Password must contain at least one digit

    Scenario: Reject letters-only password
      Given the password field contains "abcdefgh"
      When the user submits the form
      Then an error "Must contain a number" should appear

10. Best Practices for Writing Gherkin

  • Write from the user's perspective — avoid UI details like "click the blue button at top right".
  • One scenario = one behaviour — do not test two different things in one scenario.
  • Keep steps short — a step should fit on one line.
  • Use present tense — "the user clicks" not "the user clicked".
  • Avoid logic in Gherkin — no if/else, no loops — that belongs in step definitions.
  • Use Background sparingly — if only 2 of 10 scenarios need the same setup, use hooks instead.

Practice Exercise

  1. Create a new file src/features/search.feature.
  2. Write a Feature block for "Product Search" with a user story.
  3. Write three scenarios:
    • Successful search with results found.
    • Search with no results.
    • Search with a special-character keyword.
  4. Add a Background step that navigates to the homepage.
  5. Tag the first scenario @smoke and the second @regression.
