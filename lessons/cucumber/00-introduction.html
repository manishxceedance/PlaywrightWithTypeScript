Lesson 0: What Is BDD and Cucumber?

Level: Beginner No prior experience required. Start here if you are completely new to BDD.

1. The Problem BDD Solves

In traditional testing, developers write code, testers write scripts separately, and the business team writes requirements in documents. These three groups rarely speak the same language — the result is tests that do not match requirements and bugs that slip through.

Behaviour Driven Development (BDD) closes this gap by using a shared, plain-English language that all three groups can read and agree on before a single line of code is written.

2. What Is BDD?

BDD is a software development practice where teams define the expected behaviour of a feature in structured English sentences called scenarios. These scenarios become automated tests.

Three pillars of BDD:

  • Discovery — Business, dev, and QA discuss features together.
  • Formulation — Scenarios are written in Gherkin before coding starts.
  • Automation — Gherkin scenarios are linked to automated test code.

3. What Is Cucumber?

Cucumber is the most popular tool for BDD automation. It reads plain-text feature files written in Gherkin syntax and executes the corresponding automation code called step definitions.

Cucumber is available for many languages. In this track you will use Cucumber-js with TypeScript.

4. What Is Gherkin?

Gherkin is a structured plain-English format that describes how a feature should behave. A simple Gherkin scenario looks like this: 

Feature: User Login

  Scenario: Successful login with valid credentials
    Given the user is on the login page
    When the user enters username "admin" and password "secret"
    Then the user should see the dashboard

The key keywords are:

  • Feature — Groups related scenarios under one topic.
  • Scenario — One specific test case described in plain English.
  • Given — Describes the starting context (preconditions).
  • When — Describes the action the user takes.
  • Then — Describes the expected outcome (assertion).
  • And / But — Continues the previous step type to avoid repetition.

5. How the Pieces Fit Together

┌──────────────────────────────────┐
│        Feature File (.feature)   │  ← Written by QA + Business
│  Gherkin scenarios in plain text │
└─────────────┬────────────────────┘
              │ matched by step text
┌─────────────▼────────────────────┐
│     Step Definitions (.ts)       │  ← Written by automation engineer
│  TypeScript functions with       │
│  Given/When/Then decorators      │
└─────────────┬────────────────────┘
              │ calls
┌─────────────▼────────────────────┐
│     Playwright / Browser API     │  ← Interacts with real browser
└──────────────────────────────────┘

6. BDD vs TDD vs Traditional Testing

Aspect Traditional TDD BDD
Who writes tests? QA engineers Developers Everyone together
Language Code only Code only Plain English + Code
Business readable? No No Yes
When tests are defined After development Before code Before code (as scenarios)

7. Why Combine Cucumber with Playwright?

Playwright gives you full control over a real browser. Cucumber gives you readable, business-aligned test scenarios. Together they produce tests that:

  • Can be reviewed by non-technical stakeholders.
  • Run real browser interactions (clicks, form fills, navigation).
  • Generate living documentation of what your application does.
  • Integrate into CI/CD pipelines with structured JSON/HTML reports.

8. Real-World Use Case

Imagine you are testing an e-commerce checkout feature:

Feature: Checkout process

  Scenario: Guest user completes a purchase
    Given a guest user has added "Laptop" to the cart
    When the user proceeds to checkout
    And enters valid shipping and payment details
    Then the order confirmation page should display order number
    And a confirmation email should be sent

This scenario is readable by the product owner, the developer, and the QA engineer. When it passes, everyone has proof the feature works as agreed.

Practice Exercise

  1. Think of a feature in any website you use (e.g., login, search, shopping cart).
  2. Write one Gherkin scenario for that feature using Given/When/Then on paper or in a text file.
  3. Identify one Given, one When, and one Then step — no code needed yet.
  4. Consider: which parts would a business owner need to change? Which parts stay the same?
