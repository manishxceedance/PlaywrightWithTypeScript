Lesson 9: Advanced Topics — Parallel Execution, API Testing, and CI/CD

Level: Advanced Scale your suite, test APIs, and integrate into a CI pipeline.

1. Parallel Execution

Cucumber-js supports parallel execution across multiple workers. Each worker runs independent scenarios simultaneously, drastically reducing total run time. 

// cucumber.json
{
  "default": {
    "parallel": 4
  }
}

Important rules for parallel execution:

  • Each worker gets its own World instance — no shared state between scenarios.
  • Each worker must launch its own browser (handled by BeforeAll/AfterAll at worker level).
  • Do not write to the same file from multiple workers simultaneously.
  • Database state can cause flakiness — use unique test data per scenario.

Safe BeforeAll with Parallel Workers

// This BeforeAll runs once per WORKER (not once globally in parallel mode)
BeforeAll(async function () {
  browser = await chromium.launch({ headless: true });
});

AfterAll(async function () {
  await browser.close();
});

2. API Testing with Playwright's Request Context

Playwright includes a built-in HTTP client (APIRequestContext) that you can use for API testing directly inside Cucumber steps — no need for separate libraries like Axios. 

// src/support/world.ts — add APIRequestContext
import { APIRequestContext, request } from '@playwright/test';

export interface CustomWorld extends World {
  browser: Browser;
  context: BrowserContext;
  page: Page;
  apiContext: APIRequestContext;
}
// src/support/hooks.ts — initialise API context
Before(async function (this: CustomWorld) {
  this.context = await browser.newContext();
  this.page = await this.context.newPage();

  this.apiContext = await request.newContext({
    baseURL: process.env.API_URL ?? 'https://api.example.com',
    extraHTTPHeaders: {
      'Accept': 'application/json',
      'Authorization': `Bearer ${process.env.API_TOKEN}`
    }
  });
});

After(async function (this: CustomWorld, scenario) {
  await this.apiContext.dispose();
  await this.context.close();
});
# src/features/api-users.feature
@api
Feature: User API

  Scenario: Create a new user via API
    When a POST request is sent to "/users" with:
      | name  | Alice Test           |
      | email | alice@example.com    |
      | role  | viewer               |
    Then the response status should be 201
    And the response body should contain "id"
// src/steps/api.steps.ts
import { When, Then } from '@cucumber/cucumber';
import { DataTable } from '@cucumber/cucumber';
import { expect } from '@playwright/test';
import { CustomWorld } from '../support/world';

let lastResponse: any;

When(
  'a POST request is sent to {string} with:',
  async function (this: CustomWorld, path: string, table: DataTable) {
    const body = table.rowsHash();
    const response = await this.apiContext.post(path, { data: body });
    lastResponse = {
      status: response.status(),
      body: await response.json()
    };
  }
);

Then('the response status should be {int}', function (expectedStatus: number) {
  expect(lastResponse.status).toBe(expectedStatus);
});

Then('the response body should contain {string}', function (key: string) {
  expect(lastResponse.body).toHaveProperty(key);
});

3. Combining UI and API in One Scenario

A common pattern is to use the API to set up test data quickly, then verify through the UI: 

@authenticated
Feature: Order Management

  Scenario: Admin can view a newly created order
    Given a new order exists in the system via API
    When the admin navigates to the Orders page
    And the admin searches for the order by ID
    Then the order row should be visible in the table
// Step: create order via API, store ID in World
Given('a new order exists in the system via API', async function (this: CustomWorld) {
  const response = await this.apiContext.post('/orders', {
    data: { product: 'Laptop', quantity: 1, customerId: 'C001' }
  });
  const body = await response.json();
  this.orderId = body.id;
});

5. GitHub Actions CI/CD Integration

# .github/workflows/cucumber-tests.yml
name: Cucumber BDD Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright browsers
        run: npx playwright install --with-deps chromium

      - name: Run tests
        run: npm test
        env:
          BASE_URL: ${{ secrets.BASE_URL }}
          TEST_USER: ${{ secrets.TEST_USER }}
          TEST_PASS: ${{ secrets.TEST_PASS }}

      - name: Upload test report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: cucumber-report
          path: reports/

6. Retry Failed Tests

For flaky tests in CI, configure retry in cucumber.json: 

{
  "ci": {
    "retry": 2
  }
}

This retries a failed scenario up to 2 more times before marking it as definitively failed. Use sparingly — retries hide real flakiness.

7. Custom Step Timeout

The default step timeout is 5000ms. For slow environments, increase it globally or per step: 

// cucumber.json
{
  "default": {
    "timeout": 30000
  }
}

// Or per step definition
import { setDefaultTimeout } from '@cucumber/cucumber';
setDefaultTimeout(30 * 1000); // 30 seconds

8. Debugging Failed Tests

Useful techniques when a test fails unexpectedly:

  • Run headed: set HEADED=true to watch the browser: 
    HEADED=true npx cucumber-js
  • Slow motion: add SLOW_MO=500 to pause 500ms between actions.
  • Playwright Inspector: add await this.page.pause() in a step to freeze execution and open the inspector.
  • Screenshot attachment: always attach screenshots on failure in the After hook.
  • Page content attachment: attach this.page.content() to see the HTML at failure time.
  • Run one scenario: use --tags "@specific-tag" to isolate a single failing scenario.

9. Accessibility Testing with Playwright + Cucumber

You can integrate axe-core accessibility checks into your BDD scenarios: 

npm install --save-dev axe-playwright
# Feature
Scenario: Login page must pass accessibility checks
  Given the user is on the login page
  Then the page should have no accessibility violations
// Step definition
import { checkA11y, injectAxe } from 'axe-playwright';

Then('the page should have no accessibility violations', async function (this: CustomWorld) {
  await injectAxe(this.page);
  await checkA11y(this.page);
});
