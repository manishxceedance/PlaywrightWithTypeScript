Module 19: Project Implementation

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to build full end-to-end framework and regression strategy.

Simple Explanation

This module combines all previous topics into a production-style implementation plan.

Real-World Example

You can create smoke tests for critical flows and regression pack for broad coverage, all running in CI with reports.

Step-by-Step Example (Code)

Milestone plan:
1) Framework skeleton
2) Page objects + utilities
3) Smoke suite
4) Regression suite
5) CI + reports
6) Reliability tuning
What each milestone means:
  1. Framework skeleton: Set up the folder structure, playwright.config.ts with baseURL, reporter, retries, and tsconfig.json with strict mode. This is the foundation everything else builds on.
  2. Page objects + utilities: Create one Page Object Model class per application page. Write utility helpers for generating unique test data, making API setup calls, and formatting dates.
  3. Smoke suite: Write 5–10 tests covering the most critical user journeys (login, key feature, logout). These should complete in under 5 minutes and run on every pull request to block bad merges.
  4. Regression suite: Expand coverage with tests tagged @regression. Group by feature area so you can run only the relevant subset when releasing a specific module.
  5. CI + reports: Connect to GitHub Actions or another CI tool. Upload the playwright-report/ folder as a downloadable artifact so anyone can inspect failures without re-running the suite.
  6. Reliability tuning: Identify flaky tests, fix their root cause (usually a locator, timing, or data dependency issue), and remove any unnecessary waitForTimeout calls. Retries are a safety net, not a fix.

What each folder and file is for:
  1. tests/smoke/ and tests/regression/ — Separate folders keep fast smoke tests and broader regression tests clearly distinct. Run them independently via --project or --grep flags.
  2. pages/login.page.ts — One Page Object Model file per application page. Contains all locators and actions for that page. Tests import this class and call readable methods instead of writing raw locator code.
  3. fixtures/test-fixtures.ts — Custom Playwright fixtures that provide pre-configured objects to tests (e.g., a logged-in page, a pre-created test booking). Tests receive these automatically via dependency injection.
  4. data/users.json — Test data separated from test logic. Update user credentials or test values here without touching any test file.
  5. utils/date.util.ts, utils/api.util.ts — Reusable helper functions: format dates, generate unique IDs, call APIs to create or clean up test data before/after tests.
  6. playwright.config.ts — Master configuration: base URL, browser projects, parallel workers, retry count, trace/screenshot settings, and reporter setup.
  7. tsconfig.json — TypeScript compiler settings. Always enable strict: true to catch type errors and prevent hidden runtime bugs.

Definition of Done for Expert-Level Project

  1. Critical smoke flows are automated and stable in CI.
  2. Regression suite is tagged and runnable by feature/release needs.
  3. All common page interactions are moved into POM classes.
  4. TypeScript strict mode is enabled and typecheck passes.
  5. Reports, screenshots, and traces are available for every failed run.
  6. Flaky test list is tracked, root cause fixed, and retries are not abused.

Beginner Notes

  1. Build the framework structure before writing tests — retrofitting conventions onto a large suite is expensive.
  2. The Definition of Done checklist is your quality bar: every item should be green before you consider the framework "production ready".
  3. Treat test code the same as production code: code reviews, naming conventions, and refactoring all apply.

Common Mistake to Avoid

Building many tests before framework standards are defined. Set conventions first.

Practice Exercise

  1. Build a mini ecommerce framework with POM and data-driven tests.
  2. Create smoke and regression tags and run both in CI.
  3. Enable strict TypeScript and fix all type errors before final submission.
  4. Record a short runbook: how to execute, debug, and triage failed tests.
