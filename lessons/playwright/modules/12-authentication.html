Module 12: Authentication

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to reuse logged-in state and reduce repeated login steps.

Simple Explanation

storageState saves cookies and local storage after login. Other tests can reuse it to start already authenticated.

Real-World Example

Instead of logging in 200 times for 200 tests, login once and reuse session file.

Step-by-Step Example (Code)

// Save auth state after successful login
await page.context().storageState({
  path: 'playwright/.auth/user.json'
});
Line-by-line explanation:
  1. await page.context().storageState({ path: '...' }) — Saves the entire browser session state to a JSON file: cookies, localStorage, and sessionStorage. This is the equivalent of a browser "remembering" you are logged in.
  2. path: 'playwright/.auth/user.json' — The file path where the session is saved. Playwright reads this file in other tests so they start already authenticated, skipping the login UI completely.
  3. How to reuse in tests: In playwright.config.ts, add use: { storageState: 'playwright/.auth/user.json' } to the project config. Every test in that project starts with an active session.
  4. Setup project pattern: Create a special setup project that runs one login test first. Other projects list it as a dependency so they only start after the auth file is created.
  5. Security note: Add playwright/.auth/ to your .gitignore to avoid committing session data. Always use test accounts, never production credentials.

Beginner Notes

  1. Login once, save the session — reuse it across all tests that need an authenticated state.
  2. Without auth reuse, 200 tests that each log in via UI could add many minutes to your suite runtime.
  3. Different roles (admin, viewer) need separate auth files — one per role.

Common Mistake to Avoid

Committing auth state files with real secrets. Use test accounts and secure storage policies.

Practice Exercise

  1. Create one auth setup test that saves storage state.
  2. Run a protected-page test using saved state.
