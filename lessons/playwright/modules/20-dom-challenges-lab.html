Module 20: DOM Challenges Lab (Frames, Shadow DOM, Alerts, Duplicate Locators)

Hands-On Lab This module is a practical workshop where you solve common real-world Playwright blockers using TypeScript.

What You Will Learn

How to handle iframe content, shadow DOM, browser dialogs, and multiple similar elements with stable assertions.

Lab Targets in Your Workspace

  1. /lessons/practice-app/grid-frame.html for iframe and grid validations.
  2. /lessons/practice-app/shadow.html for shadow DOM interaction.
  3. /lessons/practice-app/grid-controls.html for duplicate controls in table rows.

1) Iframe Handling with Assertions

import { test, expect } from '@playwright/test';

test('fill mini form inside iframe', async ({ page }) => {
  await page.goto('/lessons/practice-app/grid-frame.html');

  const miniForm = page.frameLocator('iframe[title="Mini Form"]');
  await miniForm.locator('#mini-text').fill('Playwright Frame Test');
  await miniForm.locator('#mini-check').check();

  await expect(miniForm.locator('#mini-text')).toHaveValue('Playwright Frame Test');
  await expect(miniForm.locator('#mini-check')).toBeChecked();
});
Iframe example explained:
  1. page.frameLocator('iframe[title="Mini Form"]') — Creates a scoped reference to the iframe. All locator calls on miniForm search exclusively inside that frame. You cannot use page.locator() to reach elements inside iframes.
  2. miniForm.locator('#mini-text').fill('...') — Interacts with an element inside the iframe exactly like a normal page element, but using the frame reference instead of page.
  3. await expect(miniForm.locator('#mini-check')).toBeChecked() — Assertion inside the iframe. Pass the frame-scoped locator to expect() — assertions work the same way inside frames.

2) Shadow DOM Handling

import { test, expect } from '@playwright/test';

test('interact with open shadow root', async ({ page }) => {
  await page.goto('/lessons/practice-app/shadow.html');

  const host = page.locator('practice-shadow');
  await host.locator('#shadow-input').fill('shadow value');
  await host.locator('#shadow-btn').click();

  await expect(host.locator('#shadow-msg')).toHaveText('You typed: shadow value');
});
Shadow DOM example explained:
  1. page.locator('practice-shadow') — Locates the shadow host element (the custom HTML element that contains the shadow root).
  2. host.locator('#shadow-input').fill('shadow value') — Playwright automatically pierces open shadow roots when you chain a locator from the host. You do not need any special API or JavaScript — just chain from host to inner element.
  3. await expect(host.locator('#shadow-msg')).toHaveText('You typed: shadow value') — Verifies the output inside the shadow root using the same chained locator pattern.
  4. Note: This works only for open shadow roots (mode: 'open'). Closed shadow roots (mode: 'closed') are intentionally inaccessible and generally should not be automated.

3) Multiple Elements with Same Property

import { test, expect } from '@playwright/test';

test('update only Bob row when many controls are similar', async ({ page }) => {
  await page.goto('/lessons/practice-app/grid-controls.html');

  const bobRow = page.locator('#practice-grid tbody tr').filter({ hasText: 'Bob' });

  await bobRow.locator('input[type="text"]').fill('Bob updated note');
  await bobRow.locator('select').selectOption('Two');
  await bobRow.locator('input[type="checkbox"]').check();

  await expect(bobRow.locator('input[type="text"]')).toHaveValue('Bob updated note');
  await expect(bobRow.locator('select')).toHaveValue('Two');
  await expect(bobRow.locator('input[type="checkbox"]')).toBeChecked();
});
Duplicate controls example explained:
  1. page.locator('#practice-grid tbody tr').filter({ hasText: 'Bob' }) — Gets all table rows and then filters to only the row that contains "Bob". This is the key pattern for tables where every row has identical controls (same labels, same input types).
  2. bobRow.locator('input[type="text"]').fill('Bob updated note') — Because bobRow is already scoped to Bob's row, this locator only finds the text input inside that row. Alice and Charlie's inputs are completely unaffected.
  3. bobRow.locator('select').selectOption('Two') — Same scoping approach for the dropdown — only the select inside Bob's row is targeted.
  4. The three expect assertions at the end confirm that Bob's row was updated correctly AND (by implication) that other rows were not accidentally modified.

4) Alert Handling Pattern

The practice app does not currently include a native alert button, so this lab uses a self-contained demo page setup.

import { test, expect } from '@playwright/test';

test('handle confirm dialog safely', async ({ page }) => {
  await page.setContent(`
    <button id="danger" onclick="confirm('Are you sure you want to continue?')">Danger</button>
  `);

  page.once('dialog', async dialog => {
    await expect(dialog.type()).toBe('confirm');
    await expect(dialog.message()).toContain('Are you sure');
    await dialog.accept();
  });

  await page.locator('#danger').click();
});
Alert handling example explained:
  1. page.once('dialog', async dialog => { ... }) — Registers a one-time listener for any browser dialog (alert, confirm, or prompt). Critical rule: register this listener BEFORE clicking the button that opens the dialog.
  2. await expect(dialog.type()).toBe('confirm') — Verifies the dialog type. Types are: 'alert', 'confirm', 'prompt', 'beforeunload'.
  3. await expect(dialog.message()).toContain('Are you sure') — Validates the dialog message text to confirm the right dialog appeared before accepting it.
  4. await dialog.accept() — Clicks the OK/Yes button. Use await dialog.dismiss() to click Cancel instead.
  5. The handler runs automatically when the click triggers the dialog. The test continues after the handler finishes.

Beginner Notes (Important Rules)

  1. Register dialog listeners before clicking the element that opens the dialog.
  2. Always scope duplicate locators by row, card, or section before acting.
  3. Use frameLocator() for iframe content, not top-level page locators.
  4. With open shadow DOM, chain from host locator to inner locator.
  5. Add assertions right after actions so failures are easier to debug.
