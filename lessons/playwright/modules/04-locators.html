Module 4: Locators

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to find page elements in a stable and human-readable way, including when many elements look similar.

Simple Explanation

Locators are the addresses of elements on a page. Good locators survive UI changes and reduce flaky failures. When multiple elements share the same text or role, narrow them using filters, container scoping, and index as last option.

Real-World Example

If button color changes, getByRole("button", { name: "Submit" }) still works, but deep CSS chains may fail.

Step-by-Step Example (Code)

// 1) Start with semantic locators
      await page.getByRole('button', { name: 'Sign in' }).click();

      // 2) If same label appears many times, scope to a section
      const billingSection = page.getByRole('region', { name: 'Billing Address' });
      await billingSection.getByLabel('Email').fill('billing@example.com');

      // 3) Use filter when multiple rows have similar text
      const userRow = page.getByRole('row').filter({ hasText: 'Asha Gupta' });
      await userRow.getByRole('button', { name: 'Edit' }).click();

      // 4) Verify uniqueness before action when needed
      await expect(page.getByRole('button', { name: 'Delete' })).toHaveCount(3);
      await page.getByRole('button', { name: 'Delete' }).nth(1).click();
Line-by-line explanation:
  1. page.getByRole('button', { name: 'Sign in' }).click() — The most stable locator style. It finds an element by its semantic role (button, link, heading, etc.) and accessible name. Works even if CSS classes or element positions change.
  2. const billingSection = page.getByRole('region', { name: 'Billing Address' }); — Scopes to a named page section. When the same label (e.g., "Email") appears in multiple sections, scoping to the parent section makes the locator unambiguous.
  3. billingSection.getByLabel('Email').fill(...) — Chains the child locator onto the section. Playwright only searches for "Email" inside the billing area, ignoring identical labels elsewhere.
  4. page.getByRole('row').filter({ hasText: 'Asha Gupta' }) — Narrows a list of matching rows to the one row that contains specific text. Much more resilient than positional selectors like nth(2).
  5. await expect(...).toHaveCount(3) — Confirms exactly 3 Delete buttons exist before using nth(). This makes your intent clear and catches unexpected UI changes.
  6. .nth(1) — Selects the second element (0-indexed). Use as a last resort when semantic narrowing is not possible.

Beginner Notes (Locator Priority Order)

  1. Start with getByRole or getByLabel — most stable.
  2. Use getByTestId if developers have added data-testid attributes.
  3. Scope to a parent section first when duplicate labels/roles exist.
  4. Use .filter({ hasText: '...' }) to narrow matching rows in lists/tables.
  5. Use nth() only as a last resort — it breaks if items are reordered.

Common Mistake to Avoid

Using positional selectors like nth-child first. Prefer role, label, parent scoping, and filter with text before using index.

Practice Exercise

  1. Refactor one brittle CSS selector to getByRole/getByLabel.
  2. Create a locator priority checklist for your team.
  3. Handle a table where three "Edit" buttons exist and click only the row for one user.
