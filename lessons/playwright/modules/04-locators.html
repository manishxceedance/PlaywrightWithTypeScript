Module 4: Locators

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to find page elements in a stable and human-readable way, including when many elements look similar.

Simple Explanation

Locators are the addresses of elements on a page. Good locators survive UI changes and reduce flaky failures. When multiple elements share the same text or role, narrow them using filters, container scoping, and index as last option.

Real-World Example

If button color changes, getByRole("button", { name: "Submit" }) still works, but deep CSS chains may fail.

Step-by-Step Example (Code)

// 1) Start with semantic locators
      await page.getByRole('button', { name: 'Sign in' }).click();

      // 2) If same label appears many times, scope to a section
      const billingSection = page.getByRole('region', { name: 'Billing Address' });
      await billingSection.getByLabel('Email').fill('billing@hrms.local');

      // 3) Use filter when multiple rows have similar text
      const userRow = page.getByRole('row').filter({ hasText: 'Asha Gupta' });
      await userRow.getByRole('button', { name: 'Edit' }).click();

      // 4) Verify uniqueness before action when needed
      await expect(page.getByRole('button', { name: 'Delete' })).toHaveCount(3);
      await page.getByRole('button', { name: 'Delete' }).nth(1).click();
Line-by-line explanation:
  1. page.getByRole('button', { name: 'Sign in' }).click() — The most stable locator style. It finds an element by its semantic role (button, link, heading, etc.) and accessible name. Works even if CSS classes or element positions change.
  2. const billingSection = page.getByRole('region', { name: 'Billing Address' }); — Scopes to a named page section. When the same label (e.g., "Email") appears in multiple sections, scoping to the parent section makes the locator unambiguous.
  3. billingSection.getByLabel('Email').fill(...) — Chains the child locator onto the section. Playwright only searches for "Email" inside the billing area, ignoring identical labels elsewhere.
  4. page.getByRole('row').filter({ hasText: 'Asha Gupta' }) — Narrows a list of matching rows to the one row that contains specific text. Much more resilient than positional selectors like nth(2).
  5. await expect(...).toHaveCount(3) — Confirms exactly 3 Delete buttons exist before using nth(). This makes your intent clear and catches unexpected UI changes.
  6. .nth(1) — Selects the second element (0-indexed). Use as a last resort when semantic narrowing is not possible.

Beginner Notes (Locator Priority Order)

  1. Start with getByRole or getByLabel — most stable.
  2. Use getByTestId if developers have added data-testid attributes.
  3. Scope to a parent section first when duplicate labels/roles exist.
  4. Use .filter({ hasText: '...' }) to narrow matching rows in lists/tables.
  5. Use nth() only as a last resort — it breaks if items are reordered.

Complete Locator Exercise: HRMS Register + Employees Lab

Use this exercise when you want one realistic place to practice nearly every locator style a beginner should know. Start on the HRMS Register page, then log in with admin/admin and continue on the Employees page.

Practice pages: HRMS Register, HRMS Login, and HRMS Employees.

Locator Types and Best Use

Locator Type When to Use It HRMS Example
getByRole() Best default for buttons, links, headings, dialogs, tables, rows, and form controls with accessible names. page.getByRole('button', { name: 'Sign In' })
getByLabel() Best for inputs, selects, date fields, sliders, and textareas connected to a visible label. page.getByLabel('First Name')
getByPlaceholder() Useful when placeholder text is stable and meaningful, but still weaker than label-based lookup. page.getByPlaceholder('Enter your username')
getByText() Good for static text, helper text, badges, and verifying that visible content is present. page.getByText('Account Information')
getByTestId() Best when the app already provides stable test IDs for automation. page.getByTestId('login-submit')
getByTitle() Use when an element has a meaningful title attribute. page.getByTitle('Show/hide password')
getByAltText() Use for images when the app exposes meaningful alt text. page.getByAltText('HRMS practice logo')
locator('css') Use for structural hooks that are not easily expressed through accessibility locators. page.locator('#step-indicator')
locator('xpath=...') Use only when there is no cleaner role, label, text, or CSS path. page.locator("xpath=//div[@id='step-indicator']")
filter({ hasText }) Use to narrow repeated rows, cards, or list items to the exact record you need. page.getByRole('row').filter({ hasText: 'Emma' })
filter({ has }) Use when a parent element must contain another specific child locator. page.getByRole('row').filter({ has: page.getByText('Engineering') })
first(), last(), nth() Use only after you confirm multiple matches are expected and order is meaningful. page.getByRole('row').nth(1)
frameLocator() Use only for elements inside an iframe. page.frameLocator('#date-picker-frame').getByTitle('Next month')

Advanced Locator Operators You Should Also Know

Operator / Pattern When to Use It Practice Example
locator().locator() Chain from a parent container to search only inside that area. page.getByTestId('filter-bar').locator('[data-testid="search-input"]')
and() Match the same element through two locator conditions at once. page.getByRole('button').and(page.getByTitle('Show/hide password'))
or() Handle alternate UI entry points or states without branching too early. page.getByTestId('nav-employees').or(page.getByTestId('btn-view-employees')).first()
filter({ hasNotText }) Exclude rows or cards containing text you do not want. page.locator('[data-testid^="emp-row-"]').filter({ hasNotText: 'Inactive' }).first()
filter({ hasNot }) Exclude parents that contain a child locator you do not want. page.locator('.form-group').filter({ hasNot: page.getByTestId('forgot-password') }).first()

Why These Pages Are Good for Practice

  • The Register page contains labels, placeholders, test IDs, visible section text, radio buttons, selects, range input, and step indicators.
  • The Login page contains role, label, placeholder, test ID, checkbox, link, title-based, and alt-text locator opportunities.
  • The Employees page contains repeated table rows, filters, buttons, navigation links, modal dialogs, and dynamic data, which is ideal for row-scoped locators.

How to think about each locator in this exercise:
  1. getByRole() should be your first choice for buttons, links, headings, dialogs, checkboxes, and rows because it matches how users and assistive tools understand the page.
  2. getByLabel() is the most natural choice for form work because it stays stable even if the input HTML changes from input to textarea or select.
  3. getByPlaceholder() is helpful for quick demos, but placeholders are still UI copy and can change, so prefer labels when both exist.
  4. getByText() is best for visible content assertions like section names, help text, toast messages, and page state text.
  5. getByTestId() is excellent for automation-heavy apps because it avoids dependency on styling classes or fragile DOM depth.
  6. getByTitle() is a niche but valid choice when the app exposes a helpful title attribute, such as the password visibility toggle on the login page.
  7. getByAltText() is ideal for real images, and the HRMS login page now gives you a direct example through the practice logo image.
  8. locator('css') is useful for structural containers like the step indicator, especially when you need to assert layout widgets rather than user-facing controls.
  9. locator('xpath=...') is included here so beginners recognize it, but it should almost never be your first design choice in Playwright.
  10. filter({ hasText }) and filter({ has }) are the correct answer for dynamic tables because they let you target a record by business meaning instead of visual position.
  11. nth(), first(), and last() are acceptable only after stronger locators have already narrowed the search as much as possible.
  12. frameLocator() belongs to the iframe case only. The HRMS register page includes a real embedded calendar, so you can practice it without inventing an artificial example.
  13. and(), or(), hasNotText, and hasNot are advanced narrowing tools. Use them when the page can appear in multiple states or when you need to exclude noisy matches instead of selecting by position.

Mini Challenges for a New Learner

  1. Replace one getByTestId() call with a stronger semantic locator such as getByRole() or getByLabel().
  2. On the Register page, fill the same data using only getByLabel() where possible and note which fields become easier to read.
  3. On the Employees page, search for a department, then use row filtering to assert that at least one matching employee is visible.
  4. Open the login page and compare getByLabel('Password'), getByPlaceholder('Enter your password'), and getByTestId('login-password'). Decide which one you would keep in a real framework and explain why.
  5. Write one bad selector using a long CSS chain, then refactor it to a stable Playwright locator.
  6. Add one assertion that uses and() or or() only when two legitimate UI paths must be supported.

Coverage Check

This lesson now covers the main Playwright locator families used in practice: role, label, placeholder, text, alt text, title, test id, CSS, XPath, frame locators, row filtering, parent-child chaining, and positional narrowing. It also calls out the advanced operators and(), or(), hasNotText, and hasNot so a new learner sees both the common path and the advanced path in one place.

React and Vue Note

Playwright also has framework-specific locator engines such as _react= and _vue=, but those are useful only when the application is actually built with React or Vue and the team has a clear reason to couple tests to component structure. For most real projects, especially beginner-friendly automation, prefer user-facing locators like getByRole(), getByLabel(), and getByTestId() first because they remain more stable across UI refactors.

Common Mistake to Avoid

Using positional selectors like nth-child first. Prefer role, label, parent scoping, and filter with text before using index.
