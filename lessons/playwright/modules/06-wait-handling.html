Module 6: Wait Handling

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to wait for real conditions instead of fixed delays.

Simple Explanation

Playwright auto-waits in many cases, but you should still wait for clear conditions like URL change or visible text.

Real-World Example

After login, the app may take 2-8 seconds. Condition-based waits adapt automatically; fixed sleep wastes time and causes flakiness.

Step-by-Step Example (Code)

await page.waitForURL('**/dashboard');
await expect(page.getByText('Welcome')).toBeVisible();
Line-by-line explanation:
  1. await page.waitForURL('**/dashboard') — Waits until the browser address bar matches the pattern **/dashboard (any URL ending with /dashboard). Use this immediately after an action that causes a redirect, such as a form submission or login.
  2. await expect(page.getByText('Welcome')).toBeVisible() — Waits until a visible element containing "Welcome" appears on the page. Playwright keeps retrying this assertion until it becomes true or the timeout (default 5 seconds) is reached. No manual sleep() needed.
  3. Why not use waitForTimeout(5000)? Fixed waits always waste time on fast machines and still fail on slow ones. Condition-based waits adapt to actual page speed.

Beginner Notes

  1. Playwright auto-waits for elements to be ready before actions like click() and fill().
  2. Use waitForURL() after redirects, and assertion-based waits (toBeVisible, toHaveText) for dynamic content.
  3. Avoid waitForTimeout() — it is a fixed sleep and causes flaky, slow tests.

Common Mistake to Avoid

Using hard waits like waitForTimeout(5000) everywhere. Prefer assertion and condition waits.

Practice Exercise

  1. Replace fixed sleep calls in one test with condition waits.
  2. Use waitForURL after form submit redirect.
