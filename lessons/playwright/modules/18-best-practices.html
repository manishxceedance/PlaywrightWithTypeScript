Module 18: Best Practices

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to keep tests readable, stable, and scalable.

Simple Explanation

Best practices reduce maintenance cost and flaky failures as suite size grows.

Real-World Example

With clear naming and reusable helpers, new team members can understand and extend tests faster.

Step-by-Step Example (Code)

// Best-practice checklist
// 1) Clear test names
// 2) Stable locators
// 3) Deterministic data
// 4) Focused assertions
What each best practice means in practice:
  1. Clear test names: Write names that describe the expected outcome. Good: test('user sees error message when password is wrong', ...). Bad: test('test2', ...). Clear names make reports self-explanatory without reading the code.
  2. Stable locators: Prefer getByRole, getByLabel, getByTestId. Avoid deep CSS class chains like .container .list:nth-child(2) .btn--primary — they break whenever a developer changes styling or restructures HTML.
  3. Deterministic data: Use fixed test accounts or generate unique data (e.g., user_${Date.now()}@test.com). Avoid shared accounts that other tests or developers may modify, causing random interference between test runs.
  4. Focused assertions: Each test should verify one specific outcome. Multiple unrelated checks in one test make it hard to identify exactly what failed. If a test has 10 assertions for 10 different features, split it into 10 tests.
  5. No fixed sleeps: Never use waitForTimeout(3000) as a substitute for real synchronization. Use waitForURL, toBeVisible, or toHaveText instead — they adapt to page speed automatically.

Beginner Notes

  1. Follow these four rules from the start — retrofitting a large suite to fix bad practices is expensive.
  2. Code-review test files the same way you review production code. Quality standards apply equally.
  3. Document your team's locator priority order and naming conventions so everyone follows them consistently.

Common Mistake to Avoid

Adding long fixed waits instead of addressing synchronization/data problems.

Practice Exercise

  1. Refactor one flaky test using better locator and wait strategy.
  2. Create a team checklist for stable test design.
