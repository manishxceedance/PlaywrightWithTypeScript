Module 8: Validation

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to validate UI behavior in meaningful ways using Playwright assertions.

Simple Explanation

Validation means confirming expected outcomes with assertions. In Playwright, assertions are auto-retrying, so they wait for UI to reach expected state before failing.

Real-World Example

After creating an order, you should verify success message plus new row in order table with correct order id.

Step-by-Step Example (Code)

// UI assertions (auto-retrying)
      await expect(page).toHaveURL(/dashboard/);
      await expect(page.getByRole('heading', { name: 'Dashboard' })).toBeVisible();
      await expect(page.locator('.total')).toContainText('100');

      // Non-UI assertion (immediate check)
      const tax = 18;
      const amount = 100;
      await expect(amount + tax).toBe(118);

      // Soft assertion lets test continue and reports at end
      await expect.soft(page.getByTestId('optional-banner')).toContainText('Offer');

Common Assertion Patterns You Should Know

await expect(locator).toBeVisible();
      await expect(locator).toBeHidden();
      await expect(locator).toHaveText('Order placed');
      await expect(locator).toHaveCount(3);
      await expect(page).toHaveTitle(/Dashboard/);
      await expect(page).toHaveURL(/orders/);

      // Useful custom failure message
      await expect(page.getByRole('button', { name: 'Pay now' }), 'Pay button should be visible before final submit').toBeVisible();
Key assertion patterns explained:
  1. await expect(page).toHaveURL(/dashboard/) — Verifies the current URL matches the pattern. Use after navigation actions (login, form submit) to confirm the redirect succeeded.
  2. await expect(...).toBeVisible() — Checks the element is visible to the user, not just present in the HTML. An element can exist in the DOM but be hidden with CSS — this assertion catches that difference.
  3. await expect(...).toContainText('100') — Checks that the element's text includes "100" somewhere. Use this for dynamic values like prices or counts that may have surrounding text.
  4. await expect(amount + tax).toBe(118) — Validates a plain TypeScript/JavaScript calculation, not a UI element. Useful when you read values from the page and want to verify the computed result.
  5. expect.soft(...) — A soft assertion records the failure but lets the test continue running. All soft failures are reported together at the end, so one soft failure does not hide others.
  6. 'Pay button should be visible before final submit' — A custom message added as the second argument to expect(). Appears in the failure report, making it clear exactly what went wrong and why it matters.

Beginner Notes

  1. Playwright assertions auto-retry until the condition is met or the timeout is reached — no manual waits needed.
  2. Strong validation means checking multiple things: URL, visible element, and the actual business data (text, count, state).
  3. Use expect.soft() for non-critical checks that should not stop the test but should still be reported.

Common Mistake to Avoid

Checking only one generic condition (like title) and assuming feature works. Use multiple assertions: URL, visibility, business data, and state change.

Practice Exercise

  1. Add URL, visibility, and value assertions in one scenario.
  2. Validate a success toast and matching row data.
  3. Add one soft assertion and one custom assertion message.
