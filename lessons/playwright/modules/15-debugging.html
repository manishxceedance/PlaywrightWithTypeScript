Module 15: Debugging

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to investigate failures using debug tools and traces.

Simple Explanation

Debug mode lets you inspect test steps interactively. Trace viewer provides timeline, screenshots, and actions for root-cause analysis.

Real-World Example

A click failed because button was hidden by a popup. Trace reveals popup appeared before click step.

Step-by-Step Example (Code)

npx playwright test --debug
npx playwright show-trace trace.zip
Line-by-line explanation:
  1. npx playwright test --debug — Opens the Playwright Inspector, a visual debugging tool. Tests pause before each action, letting you step forward one action at a time, inspect the current page state, and try locator expressions interactively in the selector bar.
  2. npx playwright show-trace trace.zip — Opens the Trace Viewer for a specific trace file. You see a full timeline showing: every action (click, fill, navigate), before-and-after screenshots for each step, console log output, and all network requests. The fastest way to understand exactly what happened in a failed test run.
  3. Debug a specific test: npx playwright test login.spec.ts --debug — Only runs and debugs that one file.
  4. Record actions as code: npx playwright codegen https://manishxceedance.github.io/PlaywrightWithTypeScript/lessons/practice-app/ — Opens a browser where your clicks and typing are automatically recorded as Playwright code. Useful for quickly generating locator examples.
  5. Screenshot on failure: In playwright.config.ts, set screenshot: 'only-on-failure' to automatically capture a screenshot whenever a test fails, even without the full trace.

Beginner Notes

  1. Always read the trace before re-running a failing test — the trace shows the exact step and screenshot at the moment of failure.
  2. Use --debug mode when you cannot figure out why a locator is not finding an element — you can try selectors live in the Inspector.
  3. The Trace Viewer is the single most powerful debugging tool in the Playwright ecosystem.

Common Mistake to Avoid

Rerunning failed tests repeatedly without reading trace evidence.

Practice Exercise

  1. Intentionally fail one test and inspect trace timeline.
  2. Add screenshots/logs around critical checkpoints.

Hands-On Lab: Record, UI Mode, Debug Mode & Locator Inspection

This lab walks you through the three most powerful Playwright developer tools — Codegen (script recorder), UI Mode, and Debug Mode — using the Practice App that ships with this training hub. You will also observe how locators change as you interact with different element types.

Step 1 — Record a Script with Codegen

npx playwright codegen opens a live browser alongside a code panel that writes test code as you click. Open a terminal in your project root and run: 

# Record against the local Practice App (serve the folder first, or use the file path)
npx playwright codegen https://manishxceedance.github.io/PlaywrightWithTypeScript/lessons/practice-app/index.html

# Or record against any public site
npx playwright codegen https://playwright.dev
What to do inside the recorder:
  1. Click a button — e.g., click the Submit button in the mini form. Watch the code panel generate page.getByRole('button', { name: 'Submit' }).click();.
  2. Fill a text box — type in the Name field. Codegen writes page.getByLabel('Name').fill('Alice'); (or getByPlaceholder if no label exists).
  3. Select a dropdown option — choose a value. Codegen writes page.getByRole('combobox').selectOption('option-value');.
  4. Check a checkbox — tick a checkbox. Codegen writes page.getByRole('checkbox', { name: 'Remember me' }).check();.
  5. Hover over an element — move your cursor over a tooltip trigger. Codegen writes page.getByRole('button', { name: 'Info' }).hover();.
  6. Press a keyboard key — press Tab or Enter. Codegen writes page.keyboard.press('Tab');.
  7. Assert text — click the "Assert" toolbar button in the Codegen window then click any element. Codegen inserts await expect(page.getByRole('heading')).toHaveText('...');.

Locator change observation: Notice how Codegen always prefers role/label/placeholder locators over XPath or CSS. When you interact with a plain <div> that has no semantic role, Codegen falls back to page.locator('div.my-class'). This is your signal that the element lacks accessibility attributes — a good opportunity to ask developers to add data-testid or ARIA roles.

Step 2 — Run Tests in UI Mode

UI Mode provides a visual test runner with a watch mode, timeline, and locator picker — all in one browser window. 

# Open UI Mode for all tests
npx playwright test --ui

# Open UI Mode for a specific spec file
npx playwright test tests/login.spec.ts --ui
What to explore in UI Mode:
  1. Run a single test — Click the green play button next to any test name in the left panel. The right panel shows the browser live as the test executes.
  2. Watch mode — Toggle the eye icon. Every time you save a spec file, the affected tests re-run automatically. This is the fastest inner-loop workflow while writing tests.
  3. Timeline & actions panel — After a test finishes, click any action in the timeline to see the before/after DOM snapshot for that exact step.
  4. Pick locator — Click the crosshair icon in the toolbar, then hover over any element in the live browser. UI Mode shows you the best locator for that element in real time, and you can copy it directly into your code.
  5. Filter tests — Use the search box to filter by test title, tag, or file name. This is useful in large suites.
  6. Observe locator changes — Use the Pick Locator tool on: a semantic element (e.g., a labelled input) → you get getByLabel; a custom control without ARIA → you get a CSS selector. This contrast shows why semantic HTML matters for test stability.

Step 3 — Run Tests in Debug Mode

Debug Mode opens the Playwright Inspector alongside the browser so you can step through each action manually. 

# Debug all tests (pauses before first action)
npx playwright test --debug

# Debug only one spec file
npx playwright test tests/login.spec.ts --debug

# Debug and keep browser open after test ends
npx playwright test --debug --headed
What to do in the Inspector:
  1. Step over actions — Click the Step over button (or press F10) to execute one Playwright action at a time. After each step, examine the browser to confirm the element state changed as expected.
  2. Live locator evaluation — In the Locator input at the bottom of the Inspector, type any locator string (e.g., getByRole('button', { name: 'Login' })). Playwright immediately highlights the matching element(s) in the browser — no test re-run needed. This is the fastest way to experiment with locators.
  3. All operation types to try in debug mode:
    • Navigation — page.goto('url') — watch the address bar update step by step.
    • Click — page.getByRole('button').click() — see the click highlight flash on screen.
    • Fill — page.getByLabel('Email').fill('test@test.com') — text appears character by character.
    • Select — page.getByRole('combobox').selectOption('value') — dropdown selection visual.
    • Check / Uncheck — page.getByRole('checkbox').check() — checkbox toggles visually.
    • Hover — page.getByText('Hover me').hover() — tooltip or CSS hover state appears.
    • Keyboard — page.keyboard.press('Enter') — form submission or dialog confirm.
    • Wait for element — page.getByRole('dialog').waitFor() — Inspector pauses until element appears.
    • Assertion — await expect(page.getByRole('heading')).toContainText('Welcome') — passes or fails visually.
    • Screenshot — await page.screenshot({ path: 'step.png' }) — file is written to disk during the step.
  4. Observe locator changes across element types:
    Element Recommended Locator Fallback Locator
    Button with visible labelgetByRole('button', { name: 'Submit' })locator('button.submit-btn')
    Text input with <label>getByLabel('Email')locator('#email-input')
    Text input with placeholder onlygetByPlaceholder('Enter email')locator('input[placeholder]')
    Link with textgetByRole('link', { name: 'Sign in' })locator('a:has-text("Sign in")')
    CheckboxgetByRole('checkbox', { name: 'Remember me' })locator('input[type="checkbox"]')
    Dropdown (select)getByRole('combobox', { name: 'Country' })locator('select#country')
    Table rowgetByRole('row').filter({ hasText: 'Alice' })locator('tr:nth-child(2)')
    Element with data-testidgetByTestId('submit-btn')locator('[data-testid="submit-btn"]')
    Custom component (no ARIA)getByTestId('custom-picker')locator('div.custom-picker')

    Key insight: Elements with semantic roles and labels always yield shorter, more resilient locators. When you see a long CSS or XPath selector in the Inspector, it is a sign that the element needs better HTML semantics.
