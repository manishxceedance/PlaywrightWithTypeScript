Module 14: Reporting

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to read reports and communicate quality status.

Simple Explanation

Reports help teams quickly identify failed tests, affected areas, and historical quality trends.

Real-World Example

In release meeting, report shows 2 failed checkout tests with trace links, making triage faster.

Step-by-Step Example (Code)

npx playwright show-report
What this does and how to use reports:
  1. npx playwright show-report — Opens the generated HTML test report in your default browser. Run this after npx playwright test to see a full visual breakdown of results.
  2. What the HTML report shows: Test names with pass/fail status, duration per test, retry attempts, error messages with stack traces, screenshots taken on failure, and a "Trace" link for deep debugging.
  3. Enable trace on failure: In playwright.config.ts set trace: 'on-first-retry'. When a test fails and is retried, a trace file is captured automatically and linked in the report.
  4. Multiple reporters: Set reporter: [['html'], ['list']] in config to get both a visual HTML report and live terminal output at the same time during a test run.
  5. CI artifact: Upload the playwright-report/ folder as a build artifact in your CI pipeline so team members can download and inspect failures without re-running tests.

Beginner Notes

  1. The HTML report is the most useful artifact from a test run — always generate and preserve it.
  2. Do not just look at pass/fail counts. Open each failed test and read the step that failed, the error message, and the screenshot.
  3. Trace files give you a full step-by-step replay of what happened — use them for any failure you cannot reproduce locally.

Common Mistake to Avoid

Looking only at pass/fail count. Always inspect failed step details and attachments.

Practice Exercise

  1. Run tests and open report to inspect failed step details.
  2. Try Allure and compare trend visibility.
