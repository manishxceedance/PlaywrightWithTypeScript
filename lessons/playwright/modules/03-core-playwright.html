Module 3: Core Playwright

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How tests are organized, how assertions validate outcomes, and how hooks/fixtures reduce repeated setup.

Simple Explanation

test() defines one scenario, describe() groups related scenarios, expect() checks results, and hooks (beforeEach/afterEach) prepare or clean test state.

Real-World Example

You can group all checkout tests in one describe block, so anyone can quickly understand that feature coverage.

Step-by-Step Example (Code)

import { test, expect } from '@playwright/test';

test.describe('Login', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('https://example.com/login');
  });

  test('valid login', async ({ page }) => {
    await expect(page).toHaveURL(/login/);
  });
});
Line-by-line explanation:
  1. test.describe('Login', () => { — Groups related tests under a shared label. In reports, all tests inside appear nested under "Login", making it easy to see feature-level coverage at a glance.
  2. test.beforeEach(async ({ page }) => { — A hook that runs before every test inside this describe block. Use it to set up a known starting state (e.g., open the login page) so each test begins consistently.
  3. await page.goto('https://example.com/login'); — Navigates to the login URL. Because this is in beforeEach, every test in this group automatically starts on the login page without repeating this line.
  4. test('valid login', async ({ page }) => { — One focused test scenario. It inherits the setup from beforeEach and only tests one specific behaviour.
  5. await expect(page).toHaveURL(/login/); — Verifies the URL contains "login". This confirms the setup step actually landed on the correct page before proceeding.

Extra Essential Topic: Hooks and Fixtures

This is one of the most important ideas for real projects. Put repeated setup in hooks or fixtures so every test stays short and clean.

import { test, expect } from '@playwright/test';

test.describe('Cart', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/');
    await page.getByRole('button', { name: 'Accept cookies' }).click();
  });

  test('add one item', async ({ page }) => {
    await page.getByRole('button', { name: 'Add to cart' }).first().click();
    await expect(page.getByTestId('cart-count')).toHaveText('1');
  });

  test('remove item', async ({ page }) => {
    await page.getByRole('button', { name: 'Add to cart' }).first().click();
    await page.getByRole('button', { name: 'Remove' }).click();
    await expect(page.getByTestId('cart-count')).toHaveText('0');
  });
});
Cart example — what each step does:
  1. test.describe('Cart', () => { — Groups the two cart tests together. If either fails, you immediately know it is a Cart-related issue.
  2. test.beforeEach navigates to the home page and dismisses the cookie banner once, so neither cart test needs to repeat those steps.
  3. await page.getByRole('button', { name: 'Add to cart' }).first().click(); — Finds the first "Add to cart" button on the page and clicks it. .first() is used because multiple products appear on the page.
  4. await expect(page.getByTestId('cart-count')).toHaveText('1'); — Confirms the cart counter now shows "1" after adding one item. getByTestId uses the data-testid attribute set by developers for test stability.
  5. The second test adds an item and then removes it, asserting the count returns to "0" — verifying the remove action works correctly.

Beginner Notes

  1. describe() groups related tests — think of it as a folder for test cases.
  2. beforeEach() runs setup steps before every test — keeps each test short and focused.
  3. expect() with auto-retrying assertions waits for the page to reach the expected state before failing.

Common Mistake to Avoid

Writing long tests with many unrelated checks. Keep one test focused on one behavior.

Practice Exercise

  1. Create one describe block with two focused tests.
  2. Add one beforeEach hook and at least two expect validations.
