Module 11: API Testing

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to call APIs directly and validate responses with reliable assertions.

Simple Explanation

Playwright can test API endpoints without opening the UI. This speeds up checks and helps create or clean test data before UI tests.

Real-World Example

Before checkout UI tests run, create a booking through API so every run starts with a known test state.

Step-by-Step Example (Code)

import { test, expect } from '@playwright/test';

test('create booking', async ({ request }) => {
	const response = await request.post('/api/bookings', {
		data: { name: 'Asha', room: 'A101' }
	});

	expect(response.ok()).toBeTruthy();
	const body = await response.json();
	expect(body.name).toBe('Asha');
});

Beginner Notes (Line by Line)

  1. test('create booking', async ({ request }) => { — Uses the request fixture instead of page. The request fixture is Playwright's built-in HTTP client — no browser window opens, making API tests very fast.
  2. await request.post('/api/bookings', { data: {...} }) — Sends an HTTP POST request to /api/bookings with a JSON payload. Playwright automatically sets the Content-Type: application/json header when you use data:.
  3. expect(response.ok()).toBeTruthy() — Checks that the HTTP status code is in the 200–299 range (i.e., the request succeeded). response.ok() returns true for any 2xx response.
  4. const body = await response.json() — Parses the response body as JSON. Always await this, as it reads the response stream asynchronously.
  5. expect(body.name).toBe('Asha') — Validates a business field in the response. Checking status code alone is not enough — a 200 response can still return wrong data.
  6. Combine with UI tests: Use an API call in beforeEach to create test data, then let the UI test verify it is displayed correctly. Faster and more reliable than creating data through the UI.

Common Mistake to Avoid

Validating status code only. Always assert key business fields in the response body too.

Practice Exercise

  1. Write one GET and one POST API test.
  2. Validate status, required fields, and value types.
  3. Use API call to create test data for one UI test.
