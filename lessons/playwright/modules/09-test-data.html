Module 9: Test Data

Beginner Friendly This module explains concepts in plain language and then gives practical examples.

What You Will Learn

How to separate data from logic and run data-driven tests.

Simple Explanation

Instead of copying test code for each user, keep one test logic and run it with multiple datasets.

Real-World Example

Login checks for admin, manager, and viewer can run from one loop using different credentials.

Step-by-Step Example (Code)

import { test, expect } from '@playwright/test';

const users = [
  { role: 'admin', email: 'admin@example.com' },
  { role: 'viewer', email: 'viewer@example.com' }
];

for (const u of users) {
  test(`login ${u.role}`, async ({ page }) => {
    await page.getByLabel('Email').fill(u.email);
  });
}
Line-by-line explanation:
  1. import { test, expect } from '@playwright/test'; — Required import at the top of every Playwright test file. test is used to define test cases and expect is used for assertions. Without this line, the test() function on the next lines would not be recognized by TypeScript.
  2. const users = [{ role: 'admin', ... }, { role: 'viewer', ... }] — An array of test user objects. Each object holds the data needed for one test variant. Adding a new user role only requires adding one line here.
  3. for (const u of users) { — Loops through every user in the array. The code inside runs once per user — Playwright generates a separate, named test for each.
  4. test(`login ${u.role}`, async ({ page }) => { — Uses a template literal to create unique test names like "login admin" and "login viewer". Descriptive names make reports immediately clear about which scenario ran.
  5. page.getByLabel('Email').fill(u.email) — Types the current user's email automatically. Because u comes from the loop, each iteration uses a different email — no code duplication.
  6. Why this matters: Without data-driven tests you would copy-paste the same test body for each user, making maintenance much harder. Change the test once and it applies to all data rows.

Beginner Notes

  1. Keep test data in arrays, JSON files, or fixtures — never hardcode credentials directly in test logic.
  2. Data-driven tests reduce code duplication: one test body covers many input combinations.
  3. Always include negative test data (wrong password, missing field) alongside happy-path data.

Common Mistake to Avoid

Hardcoding data in many places. Put data in fixtures/files for easier maintenance.

Practice Exercise

  1. Create login tests for three roles from one dataset.
  2. Add negative test data for invalid credentials.
