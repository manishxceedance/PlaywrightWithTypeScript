Lesson 4: Interfaces and Type Aliases

Level: Intermediate Learn how to describe the shape of your data in TypeScript, so your code is clear and safe—even if you’ve never seen these terms before.

In TypeScript, you use interfaces and type aliases to tell the computer what your data should look like. This helps you catch mistakes early and makes your code easier to understand.

1. Interface Basics

An interface describes the structure of an object—what properties it must have and their types. Think of it as a contract: "Any object of this type must have these fields." 

// This interface describes a user story in a test management tool.
interface UserStory {
  id: string;              // unique identifier
  title: string;           // short description
  sprint: number;          // which sprint it belongs to
  isReadyForTest: boolean; // is it ready for QA?
}

// Example usage:
const story: UserStory = {
  id: "US-101",
  title: "Login page",
  sprint: 3,
  isReadyForTest: true
};

2. Type Alias Basics

A type alias is like giving a nickname to a type. It helps you avoid repeating yourself and makes your code easier to read.

// Allowed environments for a test run
type Environment = "dev" | "qa" | "staging" | "prod";
let env: Environment = "qa"; // Only these four values are allowed
// env = "production"; // Error: not allowed

// Build information for a release
type BuildInfo = {
  version: string;
  environment: Environment;
};
const build: BuildInfo = {
  version: "1.2.3",
  environment: "qa"
};

Real-world analogy: Think of a type alias like a label on a folder. Instead of writing out everything inside the folder every time, you just use the label.

3. Extending Interfaces (Inheritance)

You can make one interface build on top of another. This is called interface inheritance and lets you reuse common fields.

// Base interface for test results.
interface BaseResult {
  caseId: string;
  status: "pass" | "fail";
}
// ApiResult adds more fields to BaseResult.
interface ApiResult extends BaseResult {
  endpoint: string;
  responseTimeMs: number;
}
const result: ApiResult = {
  caseId: "TC-1",
  status: "pass",
  endpoint: "/api/health",
  responseTimeMs: 120
};

4. Combining Types with Intersections

You can combine multiple types into one using intersection types (the & symbol). This is useful for adding metadata or extra info.

// Audit info type.
type AuditMeta = {
  executedBy: string;
  executedAt: string;
};
// Combine ApiResult and AuditMeta into one type.
type AuditedApiResult = ApiResult & AuditMeta;
const audited: AuditedApiResult = {
  caseId: "TC-1",
  status: "pass",
  endpoint: "/api/health",
  responseTimeMs: 120,
  executedBy: "Neha",
  executedAt: "2026-05-27"
};

5. Readonly and ReadonlyArray

readonly makes a property or array element unchangeable after it’s set. This helps prevent bugs by making sure some values never change.

// Readonly property in an interface
interface Config {
  readonly apiUrl: string;
}
const cfg: Config = { apiUrl: "https://api.example.com" };
// cfg.apiUrl = "..."; // Error: cannot assign to readonly property

// Readonly array
const nums: ReadonlyArray = [1, 2, 3];
// nums[0] = 99; // Error: cannot modify
  • Use readonly for config, constants, or anything that should never change after creation.
  • Use ReadonlyArray to prevent accidental changes to arrays.

6. When to Use Interface vs Type Alias?

  • Use interface when you are describing the shape of an object (like a contract for a function or class).
  • Use type alias for unions ("a" | "b"), intersections (A & B), or when you want to combine types.
  • Pick clear, domain-specific names: TestCase, ApiContract, BugPayload.
